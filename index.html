基于 VEH 的运行时代码段加密保护

发表于 2026-04-06 分类于 技术

前言

软件保护是逆向工程对抗中的重要课题。静态分析工具（如 IDA Pro）的强大之处在于它能在程序未运行时就反汇编全部代码。如果我们能在程序运行前将 .text 段（代码段）加密，只在实际执行到某一页时才解密，就能大幅增加静态分析的难度。

本文分析一种基于 VEH（Vectored Exception Handling） 和 内存页保护 的运行时 .text 段加密方案。其核心思想是：

  1. 随机生成一字节密钥
  2. 注册一个处理 EXCEPTION_ACCESS_VIOLATION 的 VEH Handler
  3. 将 .text 段逐页加密，并设置内存保护属性为 PAGE_NOACCESS
  4. 程序继续运行，当执行流触及加密页时触发异常
  5. Handler 捕获异常，解密对应内存页，恢复 PAGE_EXECUTE_READ 属性
  6. 异常返回 EXCEPTION_CONTINUE_EXECUTION，程序无感知地继续执行

这是一种 按需解密（Decrypt-on-Demand） 的保护策略。

整体流程

关键数据结构与变量

加密密钥放在 .0dev 自定义段中，这个段具有 execute, read, write 属性。所有保护逻辑代码也编译到这个段中，从而避免”自己加密自己”导致死锁。

逐步分析

1. 获取 PE 节区信息

标准的 PE 解析流程：

  • 通过 GetModuleHandleA(0) 获取当前模块基址
  • 从 DOS Header 的 e_lfanew 定位到 NT Headers
  • 遍历 Section Header 数组，按名称匹配目标节区

这里传入 ".text" 来定位代码段。

2. 加密节区

逐页处理 .text 段：

  1. 计算有效页数：VirtualSize / 0x1000，不足一页的尾部跳过
  2. 修改内存保护：先设为 PAGE_EXECUTE_READWRITE 以允许写入
  3. 逐字节加密：对每个字节执行 _rotr8((byte + 0x10) ^ key, 69)
    • 先加 0x10
    • 再与密钥异或
    • 最后循环右移 69 位（等价于右移 69 % 8 = 5 位）
  4. 设为不可访问：加密完成后将页面设为 PAGE_NOACCESS

加密算法虽然简单（单字节密钥 + 简单位运算），但由于密钥在运行时随机生成，静态分析工具无法预知密钥值，且每次运行密钥不同。

3. 异常处理器（核心）

这是整个方案的核心。当程序访问 PAGE_NOACCESS 的页面时，Windows 会抛出 EXCEPTION_ACCESS_VIOLATION，VEH Handler 按以下流程处理：

3.1 获取触发异常的页面地址

ExceptionInformation[1] 是引发访问违规的内存地址。通过对齐到 0x1000（4KB 页边界）得到页面起始地址。

3.2 RIP 合法性验证

检查触发异常的指令地址（RIP）是否位于一个合法加载的模块内。如果有人试图从注入的 shellcode 或非法内存区域强制触发异常来解密页面，这个检查会阻止解密并让程序崩溃（EXCEPTION_CONTINUE_SEARCH 最终会导致未处理异常终止进程）。

3.3 解密——加密的逆运算

加密过程：_rotr8((byte + 0x10) ^ key, 69)
解密过程：(_rotl8(byte, 69) ^ key) - 0x10

即：先左移（逆右移）→ 异或密钥 → 减去 0x10，完美逆运算。

3.4 恢复执行属性

解密后将页面恢复为 PAGE_EXECUTE_READ，然后返回 EXCEPTION_CONTINUE_EXECUTION，CPU 会重新执行触发异常的那条指令，此时页面已经解密且可执行，程序正常继续。

4. RIP 合法性校验

通过 PEB（Process Environment Block）遍历所有已加载模块：

  • __readgsqword(0x60) 在 x64 下读取 PEB 地址
  • 遍历 InMemoryOrderModuleList 双向链表
  • 检查 RIP 是否落在某个合法模块的 [DllBase, DllBase + SizeOfImage] 范围内

这是一种反调试/反注入措施：只有来自已知模块的代码才能触发解密。

5. 初始化与自毁

关键步骤：

  1. 生成密钥：rand() % 255 + 1，确保密钥非零（避免异或无效）
  2. 注册 VEH Handler：参数 1 表示插入到 Handler 链的最前面，确保优先处理
  3. 加密 .text 段：调用前面分析的 encrypt_section
  4. 自毁加密逻辑：将 encrypt_section 到 rip_in_legit_module 之间的代码清零。这样即使攻击者找到了加密函数的位置，也无法逆推加密逻辑。由于这些函数在 .0dev 段（可写），所以可以被直接覆写
  5. 功能验证：通过读取和调用 .text 中的 test_func 来验证解密机制工作正常

为什么保护逻辑放在独立段？

这是至关重要的设计。如果保护逻辑也在 .text 段，那 encrypt_section 在加密 .text 的过程中会把自己也加密掉，导致当前正在执行的代码变成密文，程序立即崩溃。

通过将保护代码放在独立的 .0dev 段（不被加密），保证了：

  • 加密过程可以正常完成
  • VEH Handler 可以随时被调用
  • 加密完成后可以自毁加密函数

安全性分析

优势

方面 描述
抗静态分析 磁盘上的 .text 段不加密，但内存中加密后无法直接 dump 出完整代码
运行时密钥 每次运行密钥不同，无法预计算
按需解密 只有被执行到的页才会解密，减少内存中明文代码的暴露面
RIP 校验 阻止从非法地址触发解密
自毁机制 加密函数使用后被清零，无法被逆向分析

局限

方面 描述
单字节密钥 只有 255 种可能，容易暴力破解
页面解密后不重加密 解密后页面变为 PAGE_EXECUTE_READ，不会再次加密。全部代码执行一遍后等价于无保护
VEH Handler 可被摘除 攻击者可以通过 RemoveVectoredExceptionHandler 移除 Handler，然后手动解密
RIP 检查可被绕过 只要从合法模块内的 ROP gadget 触发异常，就能通过检查
性能开销 每个首次访问的页面都需要异常处理 + 整页解密，存在性能损耗

改进思路

  1. 重加密机制：解密执行后可以在下一次页面切换时重新加密上一页，保持”同时只有一页明文”的状态
  2. 更强的加密算法：使用 AES 等块加密替代简单的异或 + 位旋转
  3. 密钥保护：将密钥存储在寄存器或 debug register 中，而非全局变量
  4. 多层保护：结合代码混淆、反调试检测、完整性校验等手段
  5. Hardware Breakpoint 检测：防止攻击者在 Handler 上设断点分析解密逻辑

总结

这段代码展示了一个基于 VEH + 内存页保护的运行时代码加密方案。它利用 Windows 异常处理机制实现了透明的按需解密——程序本身不需要显式调用任何解密函数，正常的代码执行流自然触发解密过程。

虽然密码学强度有限，但这种 异常驱动的解密架构 是一种经典且实用的软件保护思路，在商业壳（如 Themida、VMProtect）中也能看到类似的设计理念。理解这种机制对于逆向工程分析和软件保护设计都很有价值。

数字签名伪造原理

发表于 2026-04-03 分类于 技术

前言

Windows 下的数字签名（Authenticode）是验证可执行文件来源和完整性的重要机制。但由于 PE 文件格式的设计特点，数字签名数据可以被从一个已签名的文件”偷取”并嫁接到另一个未签名的文件上。本文将通过分析一段签名偷取工具的源码，深入剖析其原理，并讨论这种伪造签名的实际效果与局限性。

PE 文件中的数字签名存储机制

IMAGE_DIRECTORY_ENTRY_SECURITY

PE 文件的数字签名信息存储在 Optional Header 的 Data Directory 数组中，索引为 IMAGE_DIRECTORY_ENTRY_SECURITY（值为 4）的条目指向签名数据。

该条目包含两个关键字段：

  • VirtualAddress：签名数据在文件中的原始偏移（注意：这是唯一一个使用文件偏移而非 RVA 的 Data Directory 条目）
  • Size：签名数据的大小

签名数据本身是一个 WIN_CERTIFICATE 结构，包含 PKCS#7 格式的 Authenticode 签名，通常被附加在 PE 文件的最末尾，不属于任何节区（Section）。

签名数据的特殊性

与其他 Data Directory 条目不同，Security Directory 有一个关键特性：

VirtualAddress 字段存储的是文件偏移（File Offset），而非 RVA（Relative Virtual Address）。

这意味着签名数据不参与内存映射，它只存在于磁盘文件中，位于所有节区数据之后。这个设计特性使得签名的”移植”成为可能——只需要复制二进制数据并修改目录条目的指向即可。

源码分析

以下是签名偷取工具的完整源码及逐段分析：

完整源码

第一阶段：读取源文件和目标文件

将两个文件完整读入内存，为后续的 PE 头解析和数据操作做准备。

第二阶段：PE 格式校验

通过检查 PE\0\0（0x4550）签名确认两个文件都是合法的 PE 文件，并校验 Machine 字段确保架构一致。

第三阶段：从源文件提取签名位置和大小

根据 Optional Header 的 Magic 字段判断是 PE32（0x10B）还是 PE32+（0x20B），然后从对应的 DataDirectory[4] 中取出签名数据的偏移和大小。

此时 sigMem + SigVirtualAddress 就指向了源文件中的 Authenticode 签名二进制数据。

第四阶段：将签名嫁接到目标文件

这是整个工具的核心操作：

  1. 找到目标文件最后一个节区的末尾位置（PointerToRawData + SizeOfRawData）
  2. 将源文件的签名二进制数据原封不动地复制到该位置
  3. 修改目标文件 PE 头的 IMAGE_DIRECTORY_ENTRY_SECURITY 条目，使其 VirtualAddress 指向新附加的签名数据，Size 设为签名数据的大小

第五阶段：写出文件

将原始目标文件重命名为备份，然后将修改后的数据写出为新文件。

为什么能”偷”签名

根本原因：签名数据是独立的二进制块

Authenticode 签名在 PE 文件中作为一个独立的二进制数据块存在，它：

  1. 不属于任何节区 —— 位于所有 Section 数据之后
  2. 通过 Data Directory 的指针定位 —— 只需修改 VirtualAddress 和 Size 即可让系统找到它
  3. 是自包含的 PKCS#7 结构 —— 包含完整的证书链和签名值

因此，只需要三步即可完成”移植”：

  1. 从源文件中按偏移和大小复制出签名的原始字节
  2. 将这些字节追加到目标文件末尾
  3. 修改目标文件 PE 头的 Security Directory 使其指向新位置

操作完成后，Windows 会认为目标文件”拥有”一个数字签名。

偷取的签名能被校验出来么？

答案是：能被严格校验发现，但在许多场景下仍然”有效”。

Authenticode 校验的两个层次

校验层次 说明 偷取签名的结果
签名存在性检查 检查 PE 文件是否包含数字签名结构 ✅ 通过
签名完整性校验 验证文件哈希是否与签名中记录的哈希一致 ❌ 失败

详细解释

1. Authenticode 哈希计算

Windows 在验证签名时，会对 PE 文件计算哈希值。计算过程中会排除以下区域：

  • PE 头中的 Checksum 字段
  • IMAGE_DIRECTORY_ENTRY_SECURITY 条目本身（VA 和 Size）
  • 签名数据本身（文件末尾的证书数据）

然后将计算出的哈希与签名中嵌入的哈希进行比对。由于目标文件的实际内容（代码段、数据段等）与源文件完全不同，哈希必然不匹配，严格校验一定会失败。

2. 为什么有时候”看起来有效”

尽管校验会失败，在以下场景中偷取的签名仍能产生效果：

  • 文件属性显示：在 Windows 资源管理器中右键查看文件属性 → 数字签名 选项卡，会显示签名信息（发布者、时间戳等），但状态会标记为”此数字签名无效”
  • 杀毒软件的宽松检查：部分杀毒软件在初筛时只检查签名是否存在以及证书是否来自知名发布者，而不做完整的哈希校验，这可能让恶意文件暂时逃过检测
  • 某些驱动加载器：早期或配置不严格的驱动加载流程可能只做浅层检查
  • 视觉欺骗：安全分析人员在快速审查文件时，看到”已签名”的标记可能降低警惕

signtool 校验结果对比

使用 signtool verify 或调用 WinVerifyTrust API 进行完整校验时，偷取的签名一定会被识别为无效。

实际效果总结

能做到的

效果 说明
显示签名信息 文件属性中会出现”数字签名”标签页
显示发布者 会展示原始签名的证书主体（如 Microsoft、Google 等）
绕过浅层检查 部分安全工具的初步筛选可能被误导
增加分析成本 让安全分析人员需要额外步骤来确认签名有效性

做不到的

限制 说明
通过完整校验 WinVerifyTrust 等 API 会返回错误
绕过驱动签名强制 Windows 内核强制签名校验（DSE）不会被欺骗
绕过 SmartScreen Windows Defender SmartScreen 会执行完整校验
获得签名者的信任等级 系统层面不会赋予该文件任何额外信任

防御与检测

  1. 始终使用 WinVerifyTrust 进行完整校验，不要仅检查签名是否存在
  2. 使用 signtool verify /pa 而非肉眼检查文件属性
  3. 对比签名中的哈希与文件实际哈希，两者不一致即为伪造
  4. 监控 Security Directory 的异常修改，比如签名偏移指向非常规位置
  5. 安全产品应当实现深度签名校验而非仅做存在性检测

总结

数字签名偷取本质上是一种PE 文件结构层面的操纵——将签名的二进制数据从一个文件复制到另一个文件，并修改 PE 头使系统能定位到该数据。它利用的是签名数据作为独立二进制块存储在文件末尾这一设计特点。

这种技术无法通过任何正规的签名校验，但在浅层检查的场景下具有一定的欺骗性。它常被恶意软件用于降低安全工具的警觉度，或者在视觉上伪装成合法软件。理解这一原理有助于安全从业者在分析中不被表面的”签名”信息所迷惑，始终坚持完整的签名校验流程。

EFT UnityPlayer 中 BattlEye 的机制分析

发表于 2026-04-02 更新于 2026-04-03 分类于 技术

概述

Escape from Tarkov（EFT）使用了一个定制版的 Unity 引擎（版本 2022.3.43f2_d42a179a7d86），其构建路径中包含 unity-cheat-proof 字样，表明这是专门为反作弊加固过的 Unity 构建版本。BattlEye 的集成逻辑直接嵌入在 UnityPlayer.dll 中，由 BSG（BattleState Games）自定义的 BSG::Guard 命名空间提供支持。

通过 IDA Pro 对 UnityPlayer.dll 进行逆向分析，发现 BattlEye 相关机制主要分为以下几个部分：

  1. BEService 服务状态检查 — 通过 Windows SCM API 验证 BattlEye 服务是否运行
  2. 定时重检机制 — 消息循环中每 60 秒异步重新验证 BEService 状态
  3. Guard 单例 — 包含 DLL 签名白名单、SHA256 哈希校验、进程/模块扫描器
  4. 库监控（Library Monitor） — 枚举进程及模块，将未验证的 DLL 信息通过 HTTPS 上报服务器

总结

EFT 的 UnityPlayer.dll 实现了一套多层次的 BattlEye 反作弊机制：

  1. 启动时检查：引擎初始化阶段即验证 BEService Windows 服务状态，未运行则直接终止
  2. 持续监控：消息循环中每 60 秒异步重检 BEService 状态，确保游戏运行期间服务不被终止

  6. 数据混淆：上报数据使用 XOR 轮转密钥混淆，增加抓包分析难度

整个机制深度嵌入到 Unity 引擎的初始化和运行流程中，而非以外部 DLL 注入的方式实现，这使得绕过更加困难。

EFT BattlEye Bypass 分析

发表于 2026-04-01 分类于 技术

概述

本文分析一个针对 Escape From Tarkov（塔科夫）的 BattlEye 反作弊 Bypass 项目——TarkovLauncherBypass。该项目以 DLL 的形式注入 BSG Launcher（游戏启动器），通过 Hook 进程创建流程，在游戏启动的关键时机完成三件事：

  1. Disabler：拦截游戏启动流程，阻止 BattlEye 加载并 Patch 反作弊校验函数
  2. Spoofer：伪造硬件标识（HWID），防止被 BattlEye 的硬件封禁机制追踪
  3. Injector：可选地注入额外 DLL（如 Metadata Dumper），并隐藏注入痕迹

整体架构以 DLL 注入到 BsgLauncher.exe 为起点，利用 Microsoft Detours 库对 Windows API 进行 Inline Hook。

模块一：Disabler —— BattlEye 禁用核心

模块二：Spoofer —— HWID 欺骗

目的

BattlEye 会采集硬件指纹（HWID）用于封禁。即使绕过了 BE 的实时保护，如果 HWID 已被标记，账号依然会被封。Spoofer 的目标是让 Launcher 和 Game 读取到的硬件信息都是伪造的。

Launcher 侧 Hook

在 Launcher 进程中 Hook 两个 API：

(1) GetSystemFirmwareTable —— SMBIOS 数据伪造

Hook KernelBase.dll!GetSystemFirmwareTable，当检测到读取 SMBIOS 表（'RSMB'）时，对返回数据中的敏感字段进行异或混淆：

  • Type 0（BIOS Information）：Vendor、BiosVersion、BiosReleaseDate
  • Type 1（System Information）：Manufacturer、ProductName、SerialNumber、UUID
  • Type 2（Baseboard Information）：Manufacturer、ProductName、SerialNumber
  • Type 4（Processor Information）：ProcessorId、SerialNumber

使用随机生成（或从 hwid.bin 文件加载）的 Key 进行确定性 XOR 变换，确保每次启动使用同一个 Key 时 HWID 一致。

(2) CWbemObject::Get —— WMI 查询结果伪造

Hook fastprox.dll 中的 CWbemObject::Get 方法。WMI（Windows Management Instrumentation）是另一个常用的硬件信息采集途径。Hook 后对所有返回的 VT_BSTR 类型字符串中的数字字符进行 XOR 变换。

模块三：Injector —— DLL 注入与隐藏

注入方式

经典的 LoadLibraryA 远程线程注入：

模块隐藏

注入完成后，通过操纵目标进程的 PEB（Process Environment Block）中的 LDR（Loader Data）链表来隐藏 DLL：

  1. 解链（Unlink）：从 InLoadOrderModuleList、InMemoryOrderModuleList、InInitializationOrderModuleList 三条双向链表中移除目标模块的节点
  2. 伪装名称：将模块名改为 X3DAudio1_7.dll，路径改为 C:\Windows\System32\X3DAudio1_7.dll
  3. 随机化元数据：DllBase 和 SizeOfImage 设为随机值
  4. 擦除 PE 头：用随机数据覆盖 DOS Header + NT Headers + Section Headers

这使得通过 EnumProcessModules 或遍历 LDR 链表无法找到注入的 DLL。

为什么这样就能 Bypass BattlEye？

BattlEye 的保护架构依赖于启动链的完整性：

这个 Bypass 的核心思路是从源头切断 BattlEye 的整条启动链：

层级 正常流程 Bypass 后
进程创建 Launcher → EscapeFromTarkov_BE.exe → 游戏 Launcher → 直接启动 EscapeFromTarkov.exe
BE 服务 BEService.exe 启动并注入 BEClient.dll 从不启动，BEClient.dll 从未加载
Unity 检查 UnityPlayer.dll 检查 BE 服务状态 Patch 为返回 true，跳过检查
游戏逻辑检查 ValidateAnticheat 异步任务验证 BE Patch MoveNext 为 ret，Task 永远不完成
HWID 采集 SMBIOS + WMI 读取真实硬件信息 Hook 返回混淆后的假数据
HWID 上报 GameAssembly.dll 中的函数上报 HWID Patch call 为 xor rax,rax，返回 NULL 跳过

关键在于：

  1. BattlEye 的用户态保护 (BEClient.dll) 根本没有加载，所以它的内存扫描、模块检测、代码完整性校验等功能全部不生效
  2. BattlEye 的内核态服务 (BEService) 也没有启动，所以驱动级的保护（如阻止非签名模块加载）同样不存在
  3. Unity 引擎和游戏逻辑层面的两道额外检查都被 Patch 掉了
  4. HWID 在 Launcher 和 Game 两侧都做了伪造，防止硬件层面的追踪

简单来说：BattlEye 从未被加载，自然也无法进行任何保护。 而游戏中检查 “BE 是否正常运行” 的代码也被 Patch，使得游戏可以照常运行而不会因为缺少反作弊而弹窗退出。

EFT Il2cpp Metadata 解密分析

发表于 2026-03-30 分类于 技术

概述

Escape from Tarkov（EFT）使用 Unity 引擎的 Il2cpp 后端编译，其 global-metadata.dat 文件包含所有 C# 类型、方法、字符串等元数据信息。BSG 对该文件实施了多层加密保护，防止被 Il2CppDumper 等工具直接解析。

本文基于对 GameAssembly.dll（版本 1.0.4.0.44026）的 IDA 逆向分析，详细还原 EFT 的 metadata 加密方案及其解密流程。

加密方案总览

EFT 的 metadata 保护由以下四层构成：

阶段 描述 关键函数 (RVA)
Phase 1 Body 比特置换解密 + Key 提取 sub_180499A10 (0x499A10)
Phase 1-Alt Shellcode 变体 sub_1804998C0 (0x4998C0)
Phase 2 Header MAP 混淆 sub_180499780 (0x499780)
Phase 3 Header 字段 MAP 查表解密 sub_18034AE80 (0x34AE80)
Phase 4 TypeDefinition 字段 Fisher-Yates 洗牌 generate_field_order + sub_1804EA960

关键全局变量

符号 RVA 说明
qword_185ACCF60 0x5ACCF60 s_GlobalMetadataHeader — metadata 基址指针
qword_185ACCE70 0x5ACCE70 metadata_key — std::map<int32_t, int32_t> header 解密映射表
qword_185ACCF68 0x5ACCF68 加密后的 header 副本指针

比特置换算法详解

本质上，这是一个 nibble 内的比特重排操作——对每个 4-bit nibble，先交换 bit0↔bit3，再交换 bit[0:1]↔bit[2:3]。这是一个自逆变换（对解密后的数据再执行一次相同操作可恢复原文）。

执行流程

  1. 从 qword_185ACCF60（metadata 指针全局变量）读取 metadata 基址
  2. header[0]（前 4 字节）= 待处理的 DWORD 数量 count
  3. header[1]（偏移 +4）= body 区域的偏移量
  4. body = metadata_base + body_offset
  5. 对 body[0..count/4] 的每个 DWORD 执行比特置换
  6. 解密后 body[0] = request_key（密钥）
  7. body[1..] = version 字符串
  8. 清零 header[0:8] 和整个 body 区域（一次性操作，不可重入）

变体 B：Shellcode 变体（sub_1804998C0，RVA 0x4998C0）

此变体使用动态生成的 shellcode 完成同样的操作：

shellcode 本身完成与变体 A 相同的比特置换操作，但以机器代码形式直接在 body 数据上执行。这是一种反静态分析技巧——IDA 无法直接分析动态生成的代码。

两个变体均会清零源数据，这意味着如果 hook 时机过晚（key 函数已执行），将无法再提取 key。

Phase 2：Header MAP 混淆（sub_180499780）

key 提取完成后，游戏立即调用 sub_180499780（RVA 0x499780）对 metadata header 进行 MAP 混淆：

核心逻辑

对 header 的前 64 个 int32 字段（256 字节，涵盖 Il2CppGlobalMetadataHeader 的全部 32 对 offset/size 字段）：

  1. 计算哈希：hash_i = transform_key_fn(i)
  2. 保存原始值：将 metadata[i] 的原始值存入 std::map，以 hash_i 为键
  3. 覆写 header：将 metadata[i] 替换为 hash_i

此后 header 中不再存储明文的 offset/size，而是存储经 transform_key_fn 变换的键值。

解密流程

所以解密需要的完整工作流程：

Step 1：DLL 注入与 il2cpp_init Hook

关键时序：必须在 AllocConsole() 等操作之前 hook il2cpp_init，因为游戏可能在 DLL 加载后极短时间内就调用 il2cpp_init。

Step 2：il2cpp_init Hook 回调

Step 3：Key 函数 Hook 捕获密钥

hook 同时覆盖了两个变体（get_meta_version_and_key 和 _alt），确保无论游戏使用哪个变体都能捕获到 key。

Step 4：等待游戏完成解密

Step 5：计算 transformed_key

由于 transform_key_fn 极其复杂（1824 字节，圈复杂度 24），直接复刻算法不可行。Dumper 选择直接调用游戏代码，这一策略使得即使游戏更新了哈希常量也无需修改 Dumper。

Step 6：通过 MAP 解密 Header

Step 7：TypeDefinition 字段逆洗牌

Step 8：输出文件

晚期注入（Keyless）后备方案

如果 Dumper 注入过晚（key 函数已执行），提供了 decrypt_without_key() 后备方案：

  1. 跳过 key 提取：因为 Phase 1 已执行且数据已清零
  2. 利用已填充的 MAP：游戏已经完成了 MAP 构建，直接使用 MAP 查表解密 header
  3. 跳过 TypeDefinition 洗牌还原：没有 key 就无法计算 transformed_key，无法生成正确的字段排列
  4. 输出部分解密的 metadata：header 已解密但 TypeDefinition 字段顺序可能不正确

总结

EFT 的 metadata 加密方案采用了多层防御策略：

  1. 比特置换 + Shellcode — 防止静态分析，两条路径增加逆向难度
  2. Hash MAP 混淆 — header 字段不再以明文存储，需通过运行时 MAP 查表
  3. 不可逆的 transform_key_fn — 极高复杂度的哈希函数阻止离线静态还原
  4. Fisher-Yates 字段洗牌 — TypeDefinition 字段布局随密钥变化，破坏结构体解析
  5. 一次性数据清零 — key 提取后立即销毁源数据，窗口期极短
  6. PPL 异步竞争 — 两个解密变体通过 when_any 竞争执行，增加 hook 难度

解密思路：通过早期 DLL 注入 + il2cpp_init 前置 hook + 直接调用游戏函数的方式可以绕过了所有保护层，是一个巧妙而实用的解决方案。
