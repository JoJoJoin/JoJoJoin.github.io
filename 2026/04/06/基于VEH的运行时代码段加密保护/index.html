基于 VEH 的运行时代码段加密保护

发表于 2026-04-06 分类于 技术

前言

软件保护是逆向工程对抗中的重要课题。静态分析工具（如 IDA Pro）的强大之处在于它能在程序未运行时就反汇编全部代码。如果我们能在程序运行前将 .text 段（代码段）加密，只在实际执行到某一页时才解密，就能大幅增加静态分析的难度。

本文分析一种基于 VEH（Vectored Exception Handling） 和 内存页保护 的运行时 .text 段加密方案。其核心思想是：

  1. 随机生成一字节密钥
  2. 注册一个处理 EXCEPTION_ACCESS_VIOLATION 的 VEH Handler
  3. 将 .text 段逐页加密，并设置内存保护属性为 PAGE_NOACCESS
  4. 程序继续运行，当执行流触及加密页时触发异常
  5. Handler 捕获异常，解密对应内存页，恢复 PAGE_EXECUTE_READ 属性
  6. 异常返回 EXCEPTION_CONTINUE_EXECUTION，程序无感知地继续执行

这是一种 按需解密（Decrypt-on-Demand） 的保护策略。

整体流程

关键数据结构与变量

加密密钥放在 .0dev 自定义段中，这个段具有 execute, read, write 属性。所有保护逻辑代码也编译到这个段中，从而避免”自己加密自己”导致死锁。

逐步分析

1. 获取 PE 节区信息

标准的 PE 解析流程：

  • 通过 GetModuleHandleA(0) 获取当前模块基址
  • 从 DOS Header 的 e_lfanew 定位到 NT Headers
  • 遍历 Section Header 数组，按名称匹配目标节区

这里传入 ".text" 来定位代码段。

2. 加密节区

逐页处理 .text 段：

  1. 计算有效页数：VirtualSize / 0x1000，不足一页的尾部跳过
  2. 修改内存保护：先设为 PAGE_EXECUTE_READWRITE 以允许写入
  3. 逐字节加密：对每个字节执行 _rotr8((byte + 0x10) ^ key, 69)
    • 先加 0x10
    • 再与密钥异或
    • 最后循环右移 69 位（等价于右移 69 % 8 = 5 位）
  4. 设为不可访问：加密完成后将页面设为 PAGE_NOACCESS

加密算法虽然简单（单字节密钥 + 简单位运算），但由于密钥在运行时随机生成，静态分析工具无法预知密钥值，且每次运行密钥不同。

3. 异常处理器（核心）

这是整个方案的核心。当程序访问 PAGE_NOACCESS 的页面时，Windows 会抛出 EXCEPTION_ACCESS_VIOLATION，VEH Handler 按以下流程处理：

3.1 获取触发异常的页面地址

ExceptionInformation[1] 是引发访问违规的内存地址。通过对齐到 0x1000（4KB 页边界）得到页面起始地址。

3.2 RIP 合法性验证

检查触发异常的指令地址（RIP）是否位于一个合法加载的模块内。如果有人试图从注入的 shellcode 或非法内存区域强制触发异常来解密页面，这个检查会阻止解密并让程序崩溃（EXCEPTION_CONTINUE_SEARCH 最终会导致未处理异常终止进程）。

3.3 解密——加密的逆运算

加密过程：_rotr8((byte + 0x10) ^ key, 69)
解密过程：(_rotl8(byte, 69) ^ key) - 0x10

即：先左移（逆右移）→ 异或密钥 → 减去 0x10，完美逆运算。

3.4 恢复执行属性

解密后将页面恢复为 PAGE_EXECUTE_READ，然后返回 EXCEPTION_CONTINUE_EXECUTION，CPU 会重新执行触发异常的那条指令，此时页面已经解密且可执行，程序正常继续。

4. RIP 合法性校验

通过 PEB（Process Environment Block）遍历所有已加载模块：

  • __readgsqword(0x60) 在 x64 下读取 PEB 地址
  • 遍历 InMemoryOrderModuleList 双向链表
  • 检查 RIP 是否落在某个合法模块的 [DllBase, DllBase + SizeOfImage] 范围内

这是一种反调试/反注入措施：只有来自已知模块的代码才能触发解密。

5. 初始化与自毁

关键步骤：

  1. 生成密钥：rand() % 255 + 1，确保密钥非零（避免异或无效）
  2. 注册 VEH Handler：参数 1 表示插入到 Handler 链的最前面，确保优先处理
  3. 加密 .text 段：调用前面分析的 encrypt_section
  4. 自毁加密逻辑：将 encrypt_section 到 rip_in_legit_module 之间的代码清零。这样即使攻击者找到了加密函数的位置，也无法逆推加密逻辑。由于这些函数在 .0dev 段（可写），所以可以被直接覆写
  5. 功能验证：通过读取和调用 .text 中的 test_func 来验证解密机制工作正常

为什么保护逻辑放在独立段？

这是至关重要的设计。如果保护逻辑也在 .text 段，那 encrypt_section 在加密 .text 的过程中会把自己也加密掉，导致当前正在执行的代码变成密文，程序立即崩溃。

通过将保护代码放在独立的 .0dev 段（不被加密），保证了：

  • 加密过程可以正常完成
  • VEH Handler 可以随时被调用
  • 加密完成后可以自毁加密函数

安全性分析

优势

方面 描述
抗静态分析 磁盘上的 .text 段不加密，但内存中加密后无法直接 dump 出完整代码
运行时密钥 每次运行密钥不同，无法预计算
按需解密 只有被执行到的页才会解密，减少内存中明文代码的暴露面
RIP 校验 阻止从非法地址触发解密
自毁机制 加密函数使用后被清零，无法被逆向分析

局限

方面 描述
单字节密钥 只有 255 种可能，容易暴力破解
页面解密后不重加密 解密后页面变为 PAGE_EXECUTE_READ，不会再次加密。全部代码执行一遍后等价于无保护
VEH Handler 可被摘除 攻击者可以通过 RemoveVectoredExceptionHandler 移除 Handler，然后手动解密
RIP 检查可被绕过 只要从合法模块内的 ROP gadget 触发异常，就能通过检查
性能开销 每个首次访问的页面都需要异常处理 + 整页解密，存在性能损耗

改进思路

  1. 重加密机制：解密执行后可以在下一次页面切换时重新加密上一页，保持”同时只有一页明文”的状态
  2. 更强的加密算法：使用 AES 等块加密替代简单的异或 + 位旋转
  3. 密钥保护：将密钥存储在寄存器或 debug register 中，而非全局变量
  4. 多层保护：结合代码混淆、反调试检测、完整性校验等手段
  5. Hardware Breakpoint 检测：防止攻击者在 Handler 上设断点分析解密逻辑

总结

这段代码展示了一个基于 VEH + 内存页保护的运行时代码加密方案。它利用 Windows 异常处理机制实现了透明的按需解密——程序本身不需要显式调用任何解密函数，正常的代码执行流自然触发解密过程。

虽然密码学强度有限，但这种 异常驱动的解密架构 是一种经典且实用的软件保护思路，在商业壳（如 Themida、VMProtect）中也能看到类似的设计理念。理解这种机制对于逆向工程分析和软件保护设计都很有价值。
