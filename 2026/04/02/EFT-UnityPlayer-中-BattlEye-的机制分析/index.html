EFT UnityPlayer 中 BattlEye 的机制分析

发表于 2026-04-02 更新于 2026-04-03 分类于 技术

概述

Escape from Tarkov（EFT）使用了一个定制版的 Unity 引擎（版本 2022.3.43f2_d42a179a7d86），其构建路径中包含 unity-cheat-proof 字样，表明这是专门为反作弊加固过的 Unity 构建版本。BattlEye 的集成逻辑直接嵌入在 UnityPlayer.dll 中，由 BSG（BattleState Games）自定义的 BSG::Guard 命名空间提供支持。

通过 IDA Pro 对 UnityPlayer.dll 进行逆向分析，发现 BattlEye 相关机制主要分为以下几个部分：

  1. BEService 服务状态检查 — 通过 Windows SCM API 验证 BattlEye 服务是否运行
  2. 定时重检机制 — 消息循环中每 60 秒异步重新验证 BEService 状态
  3. Guard 单例 — 包含 DLL 签名白名单、SHA256 哈希校验、进程/模块扫描器
  4. 库监控（Library Monitor） — 枚举进程及模块，将未验证的 DLL 信息通过 HTTPS 上报服务器

模块链表节点结构

Guard 单例布局 (104 字节)

一、模块注册机制

在 CRT 初始化阶段，BattlEye 相关逻辑通过 sub_1808D2640 注册到一个全局链表中。每个节点包含初始化函数和清理函数指针，在 Unity 引擎启动时按优先级执行。

BEService 模块注册 — sub_180028A20

  • 初始化函数 sub_180E45CE0：分配 16 字节内存，设置 type = 10，存储到 qword_181A23C00
  • 清理函数 sub_180E45CA0：释放 qword_181A23C00 指向的内存并置 NULL

Guard 单例注册 — sub_180028D40

同样通过 sub_1808D2640 注册，初始化函数为 sub_180E4E550，负责创建整个 Guard 反作弊框架。

二、BEService 服务状态检查

核心函数 — sub_180491E60

此函数是 BattlEye 服务检查的核心逻辑，通过 Windows 服务控制管理器 (SCM) API 验证 BEService 是否正在运行。

错误处理函数 sub_180311C40：弹出错误对话框（DialogBoxParamW 或 MessageBoxA），然后调用 exit(error_code) 终止进程。

错误码含义：

错误码 含义
33 系统错误，无法连接服务控制管理器
34 BEService 服务不存在或未安装
35 BEService 服务未运行

四、消息循环中的定时重检

消息循环 — sub_180E44B60

Unity 的 Windows 消息循环中嵌入了 BEService 的定时重新检测逻辑，每隔 60 秒 异步重检一次。

sub_180E427C0 类似于 std::async，创建一个异步任务对象（包含 mutex、condition_variable、引用计数），在后台线程中执行 sub_180491E60。这意味着 BEService 的检查不会阻塞主线程。

5.3 进程/模块扫描器 — sub_18029F410

六、库监控（Library Monitor）

7.2 RTTI 类结构

通过 RTTI 字符串识别出的关键类：

  • BSG::Guard::BattlEyeService — BattlEye 服务管理器
  • BSG::Guard::LibraryMonitor — DLL/模块监控器
  • std::_Fake_no_copy_callable_adapter — 用于异步回调的标准库适配器

7.3 代码签名验证

引擎使用 WinVerifyTrust API 进行代码签名验证，结合内置的白名单对比签名者信息。未签名或签名不匹配的 DLL 将被上报至服务器。

7.4 GameAssembly.dll 版本校验

sub_180A845B0 在加载 IL2CPP 运行时前，会验证 GameAssembly.dll 的存在性以及 il2cpp_data/etc 目录的完整性。验证失败会弹出错误 “Unable to check the client version” 并退出。

总结

EFT 的 UnityPlayer.dll 实现了一套多层次的 BattlEye 反作弊机制：

  1. 启动时检查：引擎初始化阶段即验证 BEService Windows 服务状态，未运行则直接终止
  2. 持续监控：消息循环中每 60 秒异步重检 BEService 状态，确保游戏运行期间服务不被终止
  3. DLL 白名单：内置 ~60 项 DLL 签名白名单，使用 WinVerifyTrust 验证加载模块的数字签名
  4. 进程枚举：通过 Toolhelp32 API 枚举进程和模块，覆盖当前进程及父进程
  5. 服务器上报：将未验证的模块信息通过 HTTPS POST 到 /client/libraries 端点，由服务器侧决定是否合法
  6. 数据混淆：上报数据使用 XOR 轮转密钥混淆，增加抓包分析难度

整个机制深度嵌入到 Unity 引擎的初始化和运行流程中，而非以外部 DLL 注入的方式实现，这使得绕过更加困难。
