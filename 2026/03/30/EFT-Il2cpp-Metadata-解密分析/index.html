EFT Il2cpp Metadata 解密分析

发表于 2026-03-30 分类于 技术

概述

Escape from Tarkov（EFT）使用 Unity 引擎的 Il2cpp 后端编译，其 global-metadata.dat 文件包含所有 C# 类型、方法、字符串等元数据信息。BSG 对该文件实施了多层加密保护，防止被 Il2CppDumper 等工具直接解析。

本文基于对 GameAssembly.dll（版本 1.0.4.0.44026）的 IDA 逆向分析，详细还原 EFT 的 metadata 加密方案及其解密流程。

加密方案总览

EFT 的 metadata 保护由以下四层构成：

阶段 描述 关键函数 (RVA)
Phase 1 Body 比特置换解密 + Key 提取 sub_180499A10 (0x499A10)
Phase 1-Alt Shellcode 变体 sub_1804998C0 (0x4998C0)
Phase 2 Header MAP 混淆 sub_180499780 (0x499780)
Phase 3 Header 字段 MAP 查表解密 sub_18034AE80 (0x34AE80)
Phase 4 TypeDefinition 字段 Fisher-Yates 洗牌 generate_field_order + sub_1804EA960

关键全局变量

符号 RVA 说明
qword_185ACCF60 0x5ACCF60 s_GlobalMetadataHeader — metadata 基址指针
qword_185ACCE70 0x5ACCE70 metadata_key — std::map<int32_t, int32_t> header 解密映射表
qword_185ACCF68 0x5ACCF68 加密后的 header 副本指针
dword_185AA9740 0x5AA9740 TypeDefinition 字段偏移表（26 条目）

比特置换算法详解

对每个 32 位值 x，变换过程为：

本质上，这是一个 nibble 内的比特重排操作——对每个 4-bit nibble，先交换 bit0↔bit3，再交换 bit[0:1]↔bit[2:3]。这是一个自逆变换（对解密后的数据再执行一次相同操作可恢复原文）。

执行流程

  1. 从 qword_185ACCF60（metadata 指针全局变量）读取 metadata 基址
  2. header[0]（前 4 字节）= 待处理的 DWORD 数量 count
  3. header[1]（偏移 +4）= body 区域的偏移量
  4. body = metadata_base + body_offset
  5. 对 body[0..count/4] 的每个 DWORD 执行比特置换
  6. 解密后 body[0] = request_key（密钥）
  7. body[1..] = version 字符串
  8. 清零 header[0:8] 和整个 body 区域（一次性操作，不可重入）

变体 B：Shellcode 变体（sub_1804998C0，RVA 0x4998C0）

此变体使用动态生成的 shellcode 完成同样的操作：

shellcode 本身完成与变体 A 相同的比特置换操作，但以机器代码形式直接在 body 数据上执行。这是一种反静态分析技巧——IDA 无法直接分析动态生成的代码。

两个变体均会清零源数据，这意味着如果 hook 时机过晚（key 函数已执行），将无法再提取 key。

Phase 2：Header MAP 混淆（sub_180499780）

key 提取完成后，游戏立即调用 sub_180499780（RVA 0x499780）对 metadata header 进行 MAP 混淆：

核心逻辑

对 header 的前 64 个 int32 字段（256 字节，涵盖 Il2CppGlobalMetadataHeader 的全部 32 对 offset/size 字段）：

  1. 计算哈希：hash_i = transform_key_fn(i)
  2. 保存原始值：将 metadata[i] 的原始值存入 std::map，以 hash_i 为键
  3. 覆写 header：将 metadata[i] 替换为 hash_i

此后 header 中不再存储明文的 offset/size，而是存储经 transform_key_fn 变换的键值。

transform_key_fn（sub_1801ABCF0，RVA 0x1ABCF0）

这是一个极其复杂的混淆哈希函数，函数体约 0x720 字节（1824 字节），包含多层嵌套的函数调用和 PPL 异步回调：

IDA 反编译显示此函数涉及 6+个不同的回调函数，通过 PPL 异步管道串联执行。由于极高的复杂度（圈复杂度 24），静态逆向几乎不可行。TarkovMetaDumper 的解决方案是直接调用游戏中的该函数。

解密流程

所以解密需要的完整工作流程：

Step 1：DLL 注入与 il2cpp_init Hook

关键时序：必须在 AllocConsole() 等操作之前 hook il2cpp_init，因为游戏可能在 DLL 加载后极短时间内就调用 il2cpp_init。

Step 2：il2cpp_init Hook 回调

Step 3：Key 函数 Hook 捕获密钥

hook 同时覆盖了两个变体（get_meta_version_and_key 和 _alt），确保无论游戏使用哪个变体都能捕获到 key。

Step 4：等待游戏完成解密

Step 5：计算 transformed_key

由于 transform_key_fn 极其复杂（1824 字节，圈复杂度 24），直接复刻算法不可行。Dumper 选择直接调用游戏代码，这一策略使得即使游戏更新了哈希常量也无需修改 Dumper。

Step 6：通过 MAP 解密 Header

Step 7：TypeDefinition 字段逆洗牌

Step 8：输出文件

晚期注入（Keyless）后备方案

如果 Dumper 注入过晚（key 函数已执行），提供了 decrypt_without_key() 后备方案：

  1. 跳过 key 提取：因为 Phase 1 已执行且数据已清零
  2. 利用已填充的 MAP：游戏已经完成了 MAP 构建，直接使用 MAP 查表解密 header
  3. 跳过 TypeDefinition 洗牌还原：没有 key 就无法计算 transformed_key，无法生成正确的字段排列
  4. 输出部分解密的 metadata：header 已解密但 TypeDefinition 字段顺序可能不正确

总结

EFT 的 metadata 加密方案采用了多层防御策略：

  1. 比特置换 + Shellcode — 防止静态分析，两条路径增加逆向难度
  2. Hash MAP 混淆 — header 字段不再以明文存储，需通过运行时 MAP 查表
  3. 不可逆的 transform_key_fn — 极高复杂度的哈希函数阻止离线静态还原
  4. Fisher-Yates 字段洗牌 — TypeDefinition 字段布局随密钥变化，破坏结构体解析
  5. 一次性数据清零 — key 提取后立即销毁源数据，窗口期极短
  6. PPL 异步竞争 — 两个解密变体通过 when_any 竞争执行，增加 hook 难度

解密思路：通过早期 DLL 注入 + il2cpp_init 前置 hook + 直接调用游戏函数的方式可以绕过了所有保护层，是一个巧妙而实用的解决方案。
